# 구조화 전환 TODO (현실 5–7일)

목표는 하나다: PoC를 “조직 자산으로 넘길 수 있는 상태”로 만드는 것.

## 전환 완료 기준 (합의)
- 같은 질문 → 같은 결과 구조
- “왜 이렇게 나왔는지” 코드/문서로 설명 가능
- 내가 없어도 다른 사람이 실행 가능

---

## DAY 1 — 경계선 고정 (기능 추가 금지)

### 1) LLM이 할 수 있는 일 / 없는 일 명시
- [ ] LLM 직접 DB 접근 완전 차단
- [ ] SQL 실행은 MCP/도구 계층으로만 가능
- [ ] 허용 쿼리 타입 정리
  - SELECT 허용
  - INSERT/UPDATE/DELETE 금지
  - JOIN 허용 테이블 목록 명시
- [ ] 위험 질문 차단 메시지 고정

## DAY 2 — 질문과 템플릿 고정

### 2) 질문 세트 파일로 고정
- [ ] `questions/daily.yaml` (일일 자동 보고서)
- [ ] `questions/deep_analysis.yaml` (심층분석 2~3개)
- [ ] 질문 문구 / 비교 기준 / 사용 지표 명시

예:
```yaml
id: daily_overview
intent: daily_summary
metrics: [inflow_pop]
comparison: previous_day
```

의도: “질문을 바꿔야 결과가 바뀐다” 상태 만들기

## DAY 3 — Semantic Layer 최소 고정

### 3) PoC 최소 Semantic Metadata 확정

  - 허용 지표 제한

## DAY 4 — 출력 표현 규칙 통일

### 4) 사람에게 보여주는 규칙 고정
- [ ] NULL / N/A 노출 금지
- [ ] PII 마스킹 시 값 삭제 금지 → 표현 전환
- [ ] 요약 문장은 “해석 → 수치” 순서 고정
- [ ] anomalies 결과 구조 고정

예:
```json
"anomalies": {
  "status": "none",
  "message": "통계적 이상 징후 없음"
}
```

UI가 아니라 JSON 계약서 고정하는 날
